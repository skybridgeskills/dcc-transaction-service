# OID4VP 1.0 verifier binding: unsigned `redirect_uri` + `direct_post` + DCQL

- Status: Accepted
- Date: 2026-07-11
- Workflow: `verify`
- Plan: `~/.skybridge/planning/dcc-transaction-service/2026-07-02-oid4vp-verifier/`

## Context

The `verify` workflow needed to accept wallet presentations over
[OpenID for Verifiable Presentations 1.0](https://openid.net/specs/openid-4-verifiable-presentations-1_0.html)
in addition to the existing VC-API / VPR / CHAPI paths, targeting the
OID4-ECDSA interop profile (Data Integrity credentials, `did:key`
holders, `ecdsa-rdfc-2019` / `eddsa-rdfc-2022`).

OID4VP 1.0 offers several axes of choice, each with a heavier and a
lighter option:

- **Query language:** DCQL (1.0-native) vs `presentation_definition` /
  PEX (pre-1.0).
- **Client identification / request integrity:** a signed request object
  under the `did:` / `x509_*` client_id schemes vs the unsigned
  `redirect_uri` prefix (§5.9.3).
- **Response delivery:** `direct_post` vs `direct_post.jwt` (encrypted /
  JARM).

The service holds **no asymmetric signing keys** (VC signing is delegated
to an external signing service; the only local JWT infra is HS256), so
signed request objects would mean net-new infrastructure. The existing
`verifier-core` pipeline already verifies Data Integrity proofs (ECDSA +
EdDSA, `did:key` + `did:web`) and runs the async Open Badges pass.

## Decision

Implement the verifier side of OID4VP 1.0 as a binding on the existing
`verify` exchange, choosing the lighter option on every axis for the MVP:

1. **DCQL only.** The DCQL query is derived from the exchange's existing
   configuration: `meta.type_values` is the **constant** VCDM
   `VerifiableCredential` IRI (per §B.1.1, `type_values` are
   fully-expanded IRIs), and `claims` map 1:1 from `vprClaims`. Specific
   credential-type / `@context` / trusted-issuer enforcement stays
   post-verification, exactly as on the VC-API path. No PEX is emitted.

2. **Unsigned request with the `redirect_uri` client_id prefix.**
   `client_id = redirect_uri:<response_uri>`; all verifier metadata
   travels inline in `client_metadata`. The request is served as unsigned
   JSON at a `request_uri` (by reference, keeping the QR small). No
   asymmetric signing infrastructure is introduced.

3. **`direct_post`** with `response_type=vp_token`. The wallet POSTs the
   DCQL `vp_token` (object keyed by credential-query id) to the
   `response_uri`.

4. **New unauthenticated per-exchange endpoints**
   (`GET .../openid4vp/request`, `POST .../openid4vp/response`), mirroring
   the OID4VCI no-auth-middleware pattern. Security is bound to the
   unguessable exchange id plus a single-use `state` (minted lazily, one
   response accepted) and the exchange `challenge` reused as the OID4VP
   `nonce`. Because `verifier-core` does not enforce the VP proof
   `domain`, the response handler enforces `domain === client_id` itself;
   `nonce` is verified cryptographically by the pipeline.

5. **Reuse the existing verify pipeline** (`preparePresentationForVerify`
   → `participateInVerifyExchange`) unchanged — the extracted VP is fed in
   as `data`, so the exchange finalizes to `complete` / `invalid` /
   `active`(async OB) identically to the VC-API path.

## Consequences

- **No new signing infra**; the binding ships as a self-contained
  `src/oid4vp/` module plus two Hono routes and a `getProtocols` entry,
  with all OID4VP runtime state inline under `variables.oid4vp`.
- **Interop contract is unsigned + DCQL.** A counterpart wallet must
  support DCQL and `direct_post`; the interop test suite's wallet driver
  currently speaks PEX only, so DCQL wallet support is a coordinated
  follow-up (not a blocker — we implement to the 1.0 spec).
- **Trust is transport-scoped.** With the `redirect_uri` prefix the wallet
  cannot cryptographically authenticate the verifier; trust rests on TLS +
  the unguessable exchange URL. Deployments needing verifier
  authentication will require the signed-request-object follow-up.
- **Deferred:** signed `did:web` / `did:key` request objects,
  `direct_post.jwt` (encrypted responses), `presentation_definition` /
  PEX, and the (non-existent in OID4VP 1.0) "authorization code flow".
  These are recorded in the plan's `future.md`, including recommended
  clarifications/errata for the OID4-ECDSA profile.

## Alternatives considered

- **Signed request object (`did:web`/`did:key` scheme).** Rejected for the
  MVP: requires net-new asymmetric signing infrastructure the service does
  not have, for a trust benefit the current interop target does not
  require.
- **`presentation_definition` / PEX.** Rejected: pre-1.0, and DCQL maps
  cleanly onto the existing `vprClaims` shape. Emitting PEX would be pure
  backwards-compatibility scaffolding.
- **A standalone verifier workflow** rather than a binding on `verify`.
  Rejected: the credential-type config, trusted-issuer/registry checks,
  claims validation, and async OB pass already live on `verify`; a binding
  reuses all of it and keeps a single result surface for polling clients.
